## Design Context

### Users

Developers managing multiple parallel Claude Code sessions across git projects. They're technical, keyboard-oriented, and expect information density. They use this alongside their terminal and editor — it's a monitoring/control surface, not the primary workspace.

### Brand Personality

**Focused, technical, calm.** A professional developer tool that communicates status clearly without being flashy. The name "cc-pewpew" adds personality but the UI itself is restrained and functional.

### Aesthetic Direction

- **Visual tone**: Modern terminal aesthetic — sleek, dark, monospace-driven
- **References**: Ghostty, Warp — clean dark UIs with precise typography and subtle accents
- **Anti-references**: Flashy dashboards, glassmorphism, gradient text, generic SaaS landing pages
- **Theme**: Dark-only. Deep navy/purple backgrounds, green accents for positive states, yellow for attention, red for errors
- **Typography**: Monospace-first (JetBrains Mono → Fira Code → Cascadia Code → Consolas). No serif or sans-serif anywhere.

### Color Palette

- Backgrounds: `#1a1a2e` (main), `#16162a` (sidebar), `#12122a` (status bar), `#0e0e1e` (terminal)
- Text: `#e0e0e0` (primary), `#9a9a9a` (secondary — 6.06:1 contrast on main bg)
- Border: `#2a2a4a`
- Accent green: `#4ade80` (running, ready, positive)
- Accent yellow: `#facc15` (needs input, attention)
- Accent red: `#f87171` (dead, error)
- Accent blue: `#60a5fa` (completed, info)

### Design Principles

1. **Status at a glance** — Session state (running/input/done) must be immediately visible from the canvas overview without clicking anything. Color, animation, and position convey meaning.
2. **Terminal-native** — The app exists to serve terminal sessions. The UI should feel like a natural extension of a terminal, not a web app wrapping one. Monospace everywhere, dark backgrounds, no rounded-corner card aesthetics beyond what's functional.
3. **Calm until urgent** — The default state is quiet and information-dense. Animation and visual emphasis are reserved for things that actually need attention (needs-input pulsing, edge indicators for off-screen sessions).
4. **Keyboard-first, mouse-friendly** — All primary actions have keyboard shortcuts. Context menus and click targets are generous but not the only way to do things.
5. **Density over whitespace** — Developers want to see many sessions at once. Cards should be compact, text small-but-readable, spacing tight. The zoomable canvas handles scale.

### Accessibility

- WCAG AA compliance (4.5:1 contrast minimum for normal text)
- Keyboard navigation for all interactive elements
- Focus-visible indicators on all interactive elements
- `aria-label` on icon-only buttons and indicators
